Architecture
Serial network architecture
The serial network architecture allows the connection of HMI/SCADA to a set of
PowerLogic P7 devices using a multi-drop RS-485 serial communication link with
client-server (master-outstation) communication.

SCADA

Serial
link

M71102A

Ethernet network architectures
The Ethernet network architecture allows the connection of a set of
PowerLogic P7 devices directly on an Ethernet network.
NOTE: It is possible to mix any of the three Ethernet protocols (IEC 61850,
DNP3, Modbus server) on the same SCADA Ethernet network.
The architecture allows the use of Generic Object Oriented Substation Event
(GOOSE) messages between devices together with another protocol for
communication to Supervisory Control and Data Acquisition (SCADA).
It is also possible to connect PowerLogic P7 devices to more than one control
system, using the same Ethernet communication port with one of the chosen
protocols.
Depending on the configuration, PowerLogic P7 devices support IEC 61850
station bus and process bus, in compliance with standards IEC 61850-6, 7-1, 7-2,
7-3, 7-4, 8-1 Edition 2.1 and IEC 61869-9.
Other supported protocols:
•

Precision Time Protocol (PTP) and Simple Network Time Protocol (SNTP) for
time synchronization

•

Hypertext Transfer Protocol Secure (HTTPS) for Web server

•

Redundancy protocols, refer to Parallel Redundancy Protocol (PRP), page
10.

•

Secured File Transfer Protocol (SFTP) for file transfer

Connections
PowerLogic P7 devices can be connected to an Ethernet switch using:
•

Port 1 RJ45 100 Mbps BASE-T copper wire.

•

Port 2/3 Small Form-factor Pluggable (SFP) modules for the redundant
connection using:
◦

RJ45 100/1000 Mbps BASE-T copper wire

◦

100 or 1000 Mbps BASE-FX singlemode fiber optic

◦

100 or 1000 Mbps BASE-FX multimode fiber optic

Parallel Redundancy Protocol (PRP)
Parallel Redundancy Protocol (PRP) transmits frames in parallel on two
independent network infrastructures: A and B.
The receiving device is in charge of discarding the second (redundant) frame once
it is received.
PRP provides a 0 ms recovery time in case of a communication failure, but this
requires a cost of two communication networks.

High-availability Seamless Redundancy (HSR)
High-availability Seamless Redundancy (HSR) is used in ring architectures.
Frames are transmitted on the ring in both directions and the receiving device
discards the redundant frames.
HSR provides a 0 ms recovery time and requires less additional network
components compared to PRP.

Rapid Spanning Tree Protocol (RSTP)
Rapid Spanning Tree Protocol (RSTP) virtually switches off all links that are not
necessary at a given time, changing the meshed topology into a tree topology.
The main advantage of RSTP is that it is a widespread solution, and works on any
network topology. On the other hand, RSTP may take a considerable time to
reconfigure the network in case of a network failure, change and restoration which
may not be a suitable solution for time-critical data exchange, for example, on
GOOSE communication.

Failover mode
Failover mode is a backup operational mode that automatically switches to a
standby Ethernet port. In the Failover mode for a PowerLogic P7, the device looks
for an active port and will use it for all traffic until that port is lost. When the active
port is lost, it will scan both ports until an active port is found. This mode is often
used with RSTP networks providing redundant access to the relay without
allowing trunk traffic to flow through the device.
The default redundancy setting is Failover mode, where either port can be used
for access.

Ethernet port configuration
You can configure the parameters for the Ethernet ports under Device
configuration > Slot A > Port 1 Ethernet or Port 2/3 Ethernet from
PowerLogic Engineering Suite. Each Ethernet port can be configured
independently. You can choose to activate or deactivate the supported
communication protocols for each individual physical port as needed.
The default IP addresses of Ethernet ports are:
•

Port 1: Default IP address is 192.168.1.100.

•

Ports 2/3: Default IP address is 192.168.0.100.
NOTE:
•

If Port 1 and Port 2/3 are configured with IP addresses within the same
network segment, the device enters safe mode and reports a physical IP
conflict. Reconfigure the affected IP addresses and reapply the settings to
resume normal operation.

•

If the configured IP address is already in use on the network, the device
reports a remote IP conflict and retains the configured IP address.

Configuring static routes
A static route is used for configuring remote access connection. Static routes can
be configured for each single interface.
NOTE: Only the physical interfaces support static routes configuration, not
VLAN interfaces.
To view the static routes:
•

From the HMI, expand > General > Device config > Slot A, go to Port 1
Ethernet or Port 2/3 Ethernet, and select the Port setting tab.

To view and configure the static routes:
1. From PowerLogic Engineering Suite, expand Device configuration > Slot A,
go to Port 1 Ethernet or Port 2/3 Ethernet.
2. Enable remote access connection by setting Gateway access to On.
3. Configure the parameters.

GOOSE communication
The IEC 61850 GOOSE (Generic Object-Oriented Substation Events) messaging
protocol enables fast, direct, and secure communication between IEDs over an
Ethernet network. It can replace traditional hardwired IED signals with multicast,
Ethernet-based station bus communications, making it suitable to transmit timesensitive, high-priority information such as breaker/isolator positions or protection
tripping commands.
GOOSE utilizes a publisher-subscriber model, where devices publish GOOSE
messages that reflect the current state of monitored data, and other devices
subscribe to these messages. A retransmission mechanism is used to ensure
reliability even in the presence of network interference.
When a change in monitored data occurs, a new GOOSE message is
instantaneously transmitted. Then, GOOSE messages with the same data are first
retransmitted at the minimum interval, then at gradually increasing intervals until
the maximum interval is reached. After that, retransmissions will continue at the
maximum interval (heartbeat) until the next data change occurs.

GOOSE performance
According to IEC 61850-5 and IEC 61850-10 Edition 2, the GOOSE performance
of PowerLogic P7 devices is compliant with Class P1 (< 3 ms).
The GOOSE publisher retransmission curve is Δt(n)=2*Δt(n-1), where Δt is the
minimum time configured by MinTime of the GoCB in the CID file. The first
GOOSE message is always published as fast as it can be, the final time interval is
the maximum time configured by MaxTime of the GoCB.
NOTE: MaxTime must be larger than 2*MinTime. Otherwise, MaxTime and
MinTime will be restored to their default values of 1 s and 10 ms, respectively.

DNP3
DNP3 communication enables PowerLogic P7 units to be connected to a
supervisor or other device featuring a DNP3 communication channel.
Communication is based on the client/server principle:
•

PowerLogic P7 is always a server device.

•

The client is the supervisor, which is another device.

The DNP3 protocol specifies the coding of data and the rules for exchanging this
data between a server device and a client device (supervision and control device
or RTU). DNP3 is an open (non-proprietary) protocol, which can be implemented
by any communicating device without any restrictions.
The DNP3 protocol is developed from the basic standards prepared by IEC
Technical Committee 57 (Power systems management and associated
communications).
Defined by IEEE Task Force C.2, DNP3 is the IEEE Recommendation for
communication between RTUs and IEDs.
For more information on the Intelligent Electronic Device protocol can be obtained
from the DNP3 User Group (www.dnp.org).

Time synchronization
The time of PowerLogic P7 corresponds to Universal Coordinated Time (UTC).
PowerLogic P7 supports time synchronization command requests and the polling
of current date and time information from a DNP3 client.
It is possible for client to verify the correctness of system time.
The function code for actioning the time synchronization command is 2 (Write).
The function code for polling the current date and time information is 1 (Read).
The time synchronization procedure for TCP and serial is different.
PowerLogic P7 devices do not retry time synchronization messages at either the
Application or Data Link layers for these application layer function codes.
•

DELAY_MEASURE request from client and corresponding response (function
code 129) from outstation

•

WRITE requests from client with an Absolute Time object, group 50, variation
1

•

WRITE requests from client with a Last Recorded Time object, group 50,
variation 3

When PowerLogic P7 detects that the time synchronization request has not been
received within the configured timeout, the IIN1.4 [NEED_TIME] bit is set in the
response message. The client must send the time synchronization request after
receiving a response with this bit set.

Modbus server
The Modbus interface is a client/server protocol defined by the Modbus
organization.
For more information on the Modbus protocol, visit www.modbus.org.
The Modbus server is used to exchange information between a client and one or
more server units, identified by a number. It implements request-reply dialog,
where requests are always initiated by the client. Modbus exists in binary (RTU
mode) formats. Data is exchanged in the form of 16-bit words (also called
registers(8)) or simply bits. Each piece of information (bit or register) has a 16-bit
address. Modbus is a data-transmission protocol in charge of communication
using serial link or Ethernet links.
Modbus TCP/IP offers the same functionality as a Modbus serial link, as well as
compatibility with multi-client architectures.
PowerLogic P7 is considered in Modbus communication as a server only.
The Modbus server supports the following functions and services:
•

Status polling

•

Measurement polling

•

Remote control

•

Time synchronization

•

Events

•

Diagnostics

•

File transfer

Modbus protocol data unit
Every Modbus request or response frame includes a Modbus PDU (protocol data
unit) made up of 2 fields.
•

Function code (1 byte): indicates the type of request (1 to 127)

•

Data (0 to n bytes): depends on the function code

The function codes in the reply and in the request are identical.

Modbus serial RTU link frames

Disturbance record extraction
The device records waveform data in COMTRADE format. Each waveform
capture is saved as one disturbance record (DR) file (.CFF).
The device can store up to 200 DR files in circular buffer. Each file has a unique
file number for retrieval. The file numbering wraps back to 1 after reaching 200.
When the storage limit is reached, the oldest file is overwritten by the newest file.

DR extraction mechanism
NOTE: In this section, record refers to a Modbus file record as defined by
Read File Record (function code 0x14), not a disturbance event record.
The device supports DR extraction with Modbus Function Code 20 (0x14). A
Modbus client extracts a DR file by sending repeated requests to retrieve DR data
in Modbus file records and reassembling the returned records into a .CFF output
file.
The device supports DR extraction by multiple clients at the same time, each
client holding its own extraction state.
Record size
The device supports requesting up to 120 registers per record. The first register
contains record metadata; the remaining registers carry user data. Each register is
2 bytes, so each record extraction response frame can contain up to 240 bytes of
data (including 2 bytes of record metadata).
Maximum file size
A file can contain up to 65,536 records (record numbers 0–65,535). Therefore, the
maximum file size is:
65,536 records × 119 registers × 2 bytes = 15,597,568 bytes = 14.875 MiB.
If the computed record number exceeds 65,535, the Modbus server rejects the
transfer for that file and returns a Modbus exception.

DR extraction procedure
To extract a DR file, the client repeats the following steps until the device indicates
end of file (EOF):
1. Build the request frame by defining file number, record number and record
length. See Request frame format, page 48.
2. Send a Read File Record request (function code 0x14) for the current record.
3. Validate the response.
•

A successful response echoes function code 0x14.

•

If the request fails, the device returns an exception response.

4. Parse the record metadata (the first returned register). See Response frame
format.
•

Read the valid-byte count (high byte).

•

Read the status flags (low byte): check the last-part flag (bit 2) and the
error flag (bit 3).

5. Append the number of bytes indicated by valid-byte count from the returned
record data bytes to the output file. Discard any trailing padding bytes.
6. Check for EOF.
•

If the last-part flag is set, finalize the current file and stop requesting
further records for this file.

•

Otherwise, increment the record number by 1 and return to step 1.

EOF detection
EOF is determined by the last-part flag (b2) in the record metadata of
a successful response.
Error handling
•

Exception response: If the device returns an exception, verify Modbus server
address, DR file number, record number, and record length.

•

Error flag (b3): If the error flag in record metadata is set, abort the current file
transfer. The device has signaled an internal error.

•

Record length consistency: Keep the record length unchanged for the
duration of the transfer unless the device specifies otherwise.

Measurands
Measurand object transmission is defined by the IEC 60870-5-103 standard.
Analog measurands are transmitted as a set of class 2 data, known as a class 2
measurands frame.
According to the standard, the coding of IEC 60870-5-103 measurand objects
must be 13-bit signed values.
For instance, when an IEC 60870-5-103 measurand (phase current) is scaled as
2.4, it means that the measurand value 1 corresponds to 2.4 × In, measurand
value 0.5 corresponds to 1.2 × In, and so on. If the measurand value in this case
exceeds 2.4 × In, the IEC 60870-5-103 object value saturates at the maximum
value and an overflow flag is set in the object transmission.

Class 2 measurands
The interchangeable part of the IEC 60870-5-103 standard specifies only five
class 2 measurands frames exist. These five class 2 frames only relate to current
and voltage values only. The allowed scale factors, meaning [max values] of per
unit coded measurands, are 1.2 or 2.4. PowerLogic P7 applies the scale factor
2.4. The device supports all five interchangeable class 2 measurand frames
defined by the standard as well as additional two private class 2 frames.
NOTE: The IEC 60870-5-103 measurand value is signed, even if the original
value is exclusively positive.

Parallel Redundancy Protocol (PRP)
Introduction
The Parallel Redundancy Protocol used in PowerLogic P7 devices is defined in
Clause 4 of the IEC 62439-3 standard. The PRP is a redundancy in the devices
method that provides bumpless switchover in case of network failure or
reintegration. Furthermore, it provides the shortest Ethernet network
reconfiguration time as network reconfiguration is seamless.
PowerLogic P7 uses two independent Ethernet ports that operate in parallel on
two independent networks. Each message is replicated and sent over both
networks. The first network node that receives a message will process it, all later
instances of the received message will be discarded. These details of replicating
and discarding messages are controlled by the low-level PRP layer of the network
architecture, so that the two networks are hidden from the higher level layers.
Thus, PRP-based networks provide a high degree of robustness and resilience.
Essentially, a PRP network consists of a pair of similar Local Area Networks
(LANs) which can be any topology (tree, ring or mesh). An example of a PRP
network is shown in PRP redundancy network, page 68.
The key features of a PRP redundant network include:

P7/EN CM/02-004A

•

Each of the two LANs can have one or more Single Attached Nodes (SANs).
These are normally non-significant devices that are attached only to a single
network. SANs can communicate with each other, but only if they are
attached to the same LAN.

•

Matched pairs of devices have an interface to each LAN, hence they are
called Dual Attached Nodes (DANs). DANs having the PRP implemented are
called DANs with PRP implemented (DANP).

•

Each device in both LANs, each DANP has to be configured with the same
Media Access Control (MAC) code and Internet Protocol (IP) address for both
of its ports. As a result, TCP/IP traffic will automatically communicate with
both of the paired devices, so it will be unaware of any layer 2 redundancy or
frame duplication issues.

•

A Redundancy Box (RedBox) is used when a single interface node has to be
connected to both LANs. The RedBox can communicate with all other nodes.
So far as other nodes are concerned, the RedBox behaves like a DAN, so an
IED connected via a RedBox is also called a Virtual DAN (VDAN). The
RedBox has its own unique IP address.

•

PowerLogic P7 devices have to be connected to the redundant Ethernet
network as a DAN using PRP (DAN using PRP is known as DANP).

•

The redundant Ethernet interface has two SFP ports. These can be fitted with
RJ45, as well as single or multi-mode LC connections. A mixture of SFP
modules is supported.

•

The management of the PRP redundancy is transparent to the application
data provided via the Ethernet interface.

67

Protection and control device

Communication protocols

•

Disconnection of one of the LANs to the device does not cause any
degradation to the application data over the Ethernet interface.

•

Each supervision frame includes a sequence number as defined in the IEC
62439-3 specification. This is incremented for each supervision message and
the value starts from zero following a system restart.

•

Supervision frames are not processed by PowerLogic P7 devices.

A SAN can be connected to any LAN and can communicate with any other SAN

not communicate directly to a SAN which is connected to the other LAN.
A DAN is connected to both LANs and can communicate with any Redundancy

communication purposes, a DAN views a SAN connected through a RedBox as a
VDAN.
When a SAN generates a basic frame, it sends the frame only onto the LAN to
which it is connected.

High-availability Seamless Redundancy (HSR)
Introduction
The High-availability Seamless Redundancy Protocol used in PowerLogic P7
devices is defined in Clause 5 of the IEC 62439-3 standard.
The HSR is a redundancy in the devices method that provides seamless
switchover and recovery in case of a single communication failure or reintegration.
HSR Ethernet redundancy method is independent of any industrial Ethernet
protocol and uses a ring topology.

PowerLogic P7 devices provide two redundant Ethernet ports using HSR. The
redundant Ethernet interface can be made using an optical fiber connection with
an LC connector type. The management of the HSR redundancy is transparent to
the application data provided via the Ethernet interface.
Disconnection of one of the nodes to the device does not cause any degradation
to the application data over the Ethernet interface.
An example of a HSR network is shown in HSR redundancy network, page 70.
The key features of a HSR redundancy network include:
•

Nodes within the ring are restricted to be HSR-capable bridging nodes, thus
avoiding the use of dedicated bridges.

•

Singly Attached Nodes (SANs) such as laptops or printers cannot be attached
directly to the ring, but need attachment through a RedBox.

•

A simple HSR network consists of doubly attached bridging nodes, each
having two ports, interconnected by full-duplex links.

•

A source Double Attached Node with HSR implemented (DANH) sends a
frame passed from its upper layers, prefixes it by an HSR tag to identify frame
duplicates and sends the frame over each port.

•

A destination DANH receives, in the fault-free state, two identical frames from
each port within a certain interval, if it is a multicast frame, it instantaneously
forwards it on the ring, removes the HSR tag of the first frame before passing
it to its upper layers and discards any duplicate.
In particular, the node will not forward a frame that it injected into the ring. A
destination node of a unicast frame does not forward a frame for which it is
the only destination, except for testing.

•

PowerLogic P7 devices have to be connected to the redundant Ethernet
network as a DAN using HSR (DANH).

The RedBox has a LRE that performs the duties of the HSR protocol, in particular:
•

since a port for maintenance often exists.

Although RSTP can recover network failures quickly, the recovery time depends
on the number of devices and the topology. The recovery time also depends on
the time taken by the devices to determine the root bridge and compute the port
roles (discarding, learning, forwarding). The devices do this by exchanging Bridge
Protocol Data Units (BPDUs) containing information about bridge devices and root
path costs. See the IEEE 802.1w standard for further information.
The RSTP solution is based on open standards. It is therefore compatible with
other manufacturers’ IEDs that use the RSTP protocol. The typical RSTP recovery
time is less than 50 ms for 10 IEDs in a network but it increases with the network
size. Due to this recovery time it is not recommended to use RSTP in automation
systems where a high availability of GOOSE or SV exchange is essential.
PowerLogic P7 devices provide two redundant Ethernet ports using RSTP. The
redundant Ethernet interface can be made using RJ45 or optical fiber connections
of LC connector type. The management of the RSTP is transparent to the
application data provided via the Ethernet interface. One of the missed node
connections to the device does not cause any degradation to the application data
over the Ethernet interface.

Failover mode
Introduction
Failover mode is a backup operational mode that automatically switches to a
standby Ethernet port. Only one of the redundant ports is active at any time. In the
Failover mode for a PowerLogic P7, the device determines the active port as
follows:
On startup, PowerLogic P7 communicates via Port 3. When the device detects a
link failure on Port 3, it activates Port 2 immediately, which means a fast port
switching (with no intentional delay) is supported.
After Failover, while communicating via Port 2, PowerLogic P7 does not check the
link status of Port 3. The device continues to communicate via the Port 2 even
after the Port 3 returns to normal. The port of choice for communications is
recalculated on failure of the Port 2 or on reboot. The device returns operation on
Port 3 under the condition if it detects a normal link status on Port 3. When the
links on Port 3 and Port 2 both fail, the device checks the link status of the two
ports in turn until it detects an active port.

VLAN and Out of Band (OOB) management
PowerLogic P7 supports VLAN and OOB on all Ethernet interfaces.

Virtual LAN (VLAN)
A virtual LAN (VLAN) is any broadcast domain that is partitioned and isolated in a
computer network at the data link layer (OSI layer 2). VLANs work through tags
within network packets and tag handling in networking systems - recreating the
appearance and functionality of network traffic that is physically on a single
network but acts as if it is split between separate networks. It is possible to assign
traffic to different VLANs to allow the separation of the traffic while using a single
physical network.
Each physical Ethernet interface supports up to 4 VLANs. VLAN tags are
accepted on all frames. VLAN can be enabled or disabled by the VLAN status
setting under Device configuration > Slot A > Port 1 Ethernet or Port 2/3
Ethernet from PowerLogic Engineering Suite.
NOTE:
•

All IP addresses must be in different network segments. Otherwise, the
device reports an IP conflict during network initialization and does not
create the VLAN.
◦

If a VLAN IP address conflicts with a physical Ethernet interface IP, the
device reports a VLAN IP conflict. The device retains the physical
interface IP and does not create the VLAN.

◦

If a VLAN IP address conflicts with another VLAN, the device reports a
VLAN IP conflict and does not create the VLAN.

•

If a configured VLAN IP address is already in use on the LAN, the device
reports a remote IP conflict and retains all configured IP addresses.

•

0.0.0.0 is considered an invalid VLAN IP address.

Out of Band (OOB)
OOB is the capability to assign IP-based services to a dedicated communication
interface. The communication interface can be a physical interface or a virtual
interface of the communication module. OOB can be configured for physical
Ethernet interfaces as well as for VLANs.
In general the Ethernet communication module is designed that all IP-based
services are working on all physical/virtual interfaces. In this case with neither
OOB nor VLAN, the communication services are available without restriction on
the physical Ethernet Interfaces. Depending on the physical module type all
services are working on Port 1 and Port 2/3.
OOB allows to configure IP-based services of networks for the customer needs.
With OOB, the user can choose which services shall work on which physical/
virtual interface. This allows, for example to configure two networks, where one
network is dedicated to the station bus and the related services and the second
network is dedicated for maintenance services. The user can configure
